                   - giving users and auditors enough info to understand why an answer was produced
				   - without transparency users cannot verify claims and auditors cannot trace decisions
				   - citations and evidence from grounded sources
				   - surface citations explicitly alongside model responses and link the claim in the output back to the source document
				   - can verify claims against original sources, and this is what makes a grounded responses
				   - an agent, grounded response is not enough - but Bedrock Agent Tracing helps
					 - Enable the tracing parameter in the API call to capture step by step agent execution
					 - Trace captures: tool invocations, knowledge base retrievals, API call made, and decision points at each reasoning step
					 - Trace data is structured and queryable for debugging, auditing, and transparency reporting
	               - Bedrock response streaming - can generate user friendly info

3. Fairness Metrics and Evaluation Datasets
  a. What fairness means for FM evaluations?
     - Fairness failures are not always obvious and you cannot improce what you do not measure
  b. Core fairness metrics (choose the metric that matches the failure mode that you and your org is most concerned for your app)
     - Fairness metrics define equal treatment
	 - Demograpgic Parity
	 - Disparate Impact DI
	 - Equal Opportunity - positive cases
	 - Equalized odds
	 - counterfactual fairness
  c. Building a representative evaluation datasets
     - Include diverse demographic perspectives and underrepresented scenarios
	 - Counterfactual test pairs use the same prompt with only demographic attribute changed, and the model's response should remain consistent
	 - Edge cases that expose model failure modes under distribution shift
	 - Apply statistical significance testing - a poorly designed evaluation dataset produces misleading fairness results

5. Automated Fairness Evaluations with LLM as a judge
   a. challenge of scale because one app is prod we cannot manually review the fairness  
      - human evaluation of fairness does not scale 
	  - rule based evaluation misses nuanced bias 
	  - llm as a judge scales fairness evaluation
   b. Implementing LLM as a judge in Amazon Bedrock
      - Use a separate foundation model as the judge 
	  - The judge returns a score and a plain-language explanation for each response evaluated
	  - Configure an evaluation job in Amazon Bedrock Model Evaluations with a designated judge model
	  - Define a fairness rubric as the evaluation
	  - Bedrock supports built-in metrics such as stereotyping and custom metrics with rating scales; pass/fail thresholds must be applied in a downstream workflow
   c. Rubric Design and Calibration
	  - A well defined rubric is what makes LLM as a Judge reliable
	  - Specific and measurable criteria
	  - Scored dimensions
	  - Ensemble judges
	  - Calibrate against human evaluations
   d. Aggregate Results and Using as Regression Gates
      - Aggregate evaluation scores across demographics
	  - Store evaluation results in s3 for audit and historical comparison
	  - Use evaluation outputs to feed automated fairness auditing workflows

6. Developing Policy-compliant AI systems
   a. A policy that list content requirements, restrainst means nothing if those are not implementing in the system that serves the user
   b. Translate policy into runtime enforcement
     - think of it as guardrails not as a safety thing but as a policy translation layer
	 - Mapping policy requirements to guardrail configurations
	   - each organizational Responsible AI policy requirement maps to a specific Guardrail filter
	   - Content safety requirements -> content filter thresholds
	   - Topic restrictions -> denied topic filters blocking responses about restricted subjects
	   - Privacy requirements -> sensitive info filters detecting and redacting PII in inputs and outputs
	   - Guardrails prevent harmful or biased language but it does not prove fairness
	   - Guardrail configuration are versioned so policy changes are retained for auditing
	 - Ensuring Guardrails are consistently attached for every invoke model API call
